## High Priority

### H-1: `uuid.MustParse` in repository Create (pre-existing, not introduced)

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/internal/modules/user/adapters/postgres/repository.go:101`

```go
ID: uuid.MustParse(string(user.ID())),
```

`MustParse` panics on invalid input. Since the UUID is generated by `domain.NewUser` via `uuid.NewString()`, it should always be valid. However, if `Reconstitute` is called with a non-UUID string (e.g., from corrupted data), the repository will panic instead of returning an error.

**Risk:** Low in practice since `NewUser` always generates valid UUIDs and `Reconstitute` is only called from `toDomain` with DB data. The existing `parseUserID` helper already uses safe `uuid.Parse` for the read paths. Consider using `uuid.Parse` here too for consistency.

**Suggested fix:**
```go
uid, err := uuid.Parse(string(user.ID()))
if err != nil {
    return fmt.Errorf("invalid user ID: %w", err)
}
_, err = q.CreateUser(ctx, sqlcgen.CreateUserParams{ID: uid, ...})
```

### H-2: `Reconstitute` in test uses non-UUID ID string

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/internal/modules/user/app/create_user_test.go:61`

```go
existing := domain.Reconstitute("existing-id", ...)
```

`"existing-id"` is not a valid UUID. This works in the current test because the ID is never parsed -- the test only checks that `ErrEmailTaken` is returned. However, if future code paths parse the returned user's ID, this will fail in confusing ways. Use a valid UUID instead.

**Suggested fix:**
```go
existing := domain.Reconstitute("00000000-0000-0000-0000-000000000001", ...)
```

### M-2: Integration tests lack `t.Parallel()`

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/internal/modules/user/adapters/postgres/repository_test.go`

Each integration test spins up its own Postgres container (via `setupRepo`). Without `t.Parallel()`, they run sequentially -- slow but safe. Adding `t.Parallel()` would speed up the suite. Not a bug, just a performance consideration for CI.

### M-3: `toDomain` passes `row.CreatedAt` / `row.UpdatedAt` as `time.Time`

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/internal/modules/user/adapters/postgres/repository.go:188-193`

The `sqlc.yaml` overrides `timestamptz -> time.Time`, so `row.CreatedAt` and `row.UpdatedAt` are `time.Time` (correct). But `DeletedAt` is `pgtype.Timestamptz` because of the nullable column. This is inconsistent -- sqlc handles nullable differently. The code handles it correctly with the `.Valid` check, but future reviewers may be confused. Consider adding a brief comment.

### M-4: `docs/adding-a-module.md` migration still uses `DEFAULT gen_random_uuid()` for `id`

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/docs/adding-a-module.md:62`

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
```

The whole point of Phase 02 was fixing the UUID mismatch by having the domain generate the UUID and passing it in the INSERT. The docs still show `DEFAULT gen_random_uuid()` on the migration, which is harmless (the default is never used if the INSERT always provides `id`), but could confuse future developers into thinking the DB generates the ID. Consider removing the `DEFAULT` or adding a comment explaining the domain owns the ID.

### M-5: `testutil.RunMigrations` does raw SQL parsing instead of using goose

**File:** `/Users/namnguyen/Desktop/www/freelance/gnha-services/internal/shared/testutil/migrate.go`

The `extractGooseUp` function does naive string splitting on `-- +goose Up` / `-- +goose Down` markers. This works for simple migrations but will break if a migration contains:
- `-- +goose StatementBegin` / `-- +goose StatementEnd` blocks (used for functions/triggers)
- Multiple `-- +goose Up` sections

This is acceptable for now (the project only has one simple migration), but document the limitation or consider using `goose.Up()` programmatically.
