## Overall Assessment

High quality scaffold generator. The CLI is defensively written, template coverage is complete, and the generated Go code will compile correctly for a plain-name module (e.g. `product`). Pattern fidelity against the user module is excellent across all layers. Six issues require attention before trusting the output for production modules.

## High Priority

### H-1: `toDomain` in postgres template expects `sqlcgen.{{.NameTitle}}` which may not exist yet

**File:** `adapter_postgres.tmpl` line 161

```go
func toDomain(row sqlcgen.{{.NameTitle}}) *domain.{{.NameTitle}} {
```

The `sqlcgen.{{.NameTitle}}` type is generated by `task generate` (sqlc codegen). Until the developer runs `task generate` after customising the migration and queries files, the generated adapter code will not compile. This is expected flow, but the "Next steps" output in `main.go` lists "Run code generation" as step 4, which is correct.

**However:** step 5 says "Update domain/app/adapters to match new fields" — the developer must also update `toDomain`, `Create{{.NameTitle}}Params`, `Update{{.NameTitle}}Params` field references in `adapter_postgres.tmpl` output. The next-steps note is vague about *which* files need updating.

**Recommendation:** Expand step 5 message to list the three concrete sites:
```
  5. Update generated adapter fields in:
       - adapters/postgres/repository.go → toDomain(), Create/Update Params
       - adapters/grpc/mapper.go → toProto()
       - domain/<name>.go → New<Name>(), Reconstitute(), getters
```

### H-2: `adapter_postgres.tmpl` uses `uuid.MustParse` in `Create` — panics on malformed ID

**File:** `adapter_postgres.tmpl` line 88

```go
ID: uuid.MustParse(string(entity.ID())),
```

The entity ID is set by `domain.New{{.NameTitle}}` via `uuid.NewString()`, so in practice this never panics. But the user module's own `Create` method does the same (line 101 of `repository.go`), so this is a known pattern in the codebase. It is consistent.

**Risk:** A future developer who calls `domain.Reconstitute` with an arbitrary string ID and then immediately calls `repo.Create` will panic at runtime with no error return. The user module carries the same risk.

**Recommendation:** Mirror the safer pattern from `parseUserID` — use `uuid.Parse` and return an error. This is a pre-existing codebase issue but worth fixing in the template so new modules don't repeat it:

```go
func (r *Pg{{.NameTitle}}Repository) Create(ctx context.Context, entity *domain.{{.NameTitle}}) error {
    uid, err := parse{{.NameTitle}}ID(entity.ID())
    if err != nil {
        return err
    }
    q := sqlcgen.New(r.pool)
    _, err = q.Create{{.NameTitle}}(ctx, sqlcgen.Create{{.NameTitle}}Params{
        ID:   uid,
        Name: entity.Name(),
    })
    ...
}
```

## Medium Priority

### M-1: `toTitle` does not handle digits — validation permits only letters/underscores

**File:** `main.go` lines 48-57, 149-157

`toTitle` splits on `_` and uppercases the first rune of each part. The validation rejects digits (`!unicode.IsLetter(r) && r != '_'`), so inputs like `order2item` are rejected at the gate. This is intentional but not documented.

If a developer ever relaxes the validation, `toTitle("order2item")` → `"Order2item"` which is valid Go but may surprise. No bug right now, just a documentation gap.

**Recommendation:** Add a comment to the validation block:
```go
// Digits are intentionally rejected: PascalCase derivation only works
// cleanly with letters and underscores.
```

### M-2: `--plural` flag input is not validated

**File:** `main.go` lines 67-71

```go
if *plural != "" {
    data.NamePlural = *plural
}
```

The plural override is used in table names (`db/migrations`), SQL (`db/queries`), package identifiers, and Go identifiers. If a developer passes `--plural "order items"` (with a space) or `--plural "OrderItems"` the output SQL and Go code will be syntactically broken.

**Recommendation:** Apply the same validation as `--name` to `--plural`:
```go
if *plural != "" {
    for _, r := range *plural {
        if !unicode.IsLetter(r) && r != '_' {
            fmt.Fprintf(os.Stderr, "error: plural must contain only letters and underscores\n")
            os.Exit(1)
        }
    }
    if *plural != strings.ToLower(*plural) {
        fmt.Fprintf(os.Stderr, "error: plural must be lowercase\n")
        os.Exit(1)
    }
    data.NamePlural = *plural
}
```

### M-3: `adapter_grpc_handler.tmpl` — response literal brace syntax is fragile

**File:** `adapter_grpc_handler.tmpl` lines 48, 56, 87

```go
return connect.NewResponse(&{{.Name}}v1.Create{{.NameTitle}}Response{ {{- .NameTitle}}: toProto(entity)}), nil
```

The `{{- .NameTitle}}` trim-whitespace directive is used to prevent a space before the field name. This renders correctly (e.g. `&productv1.CreateProductResponse{Product: toProto(entity)}`). However the intent is non-obvious and the inline template syntax is harder to read and maintain.

The user module's actual handler writes:
```go
return connect.NewResponse(&userv1.CreateUserResponse{User: toProto(user)}), nil
```

The template equivalent is functionally correct but visually unusual. No compile bug.

**Recommendation:** Use a helper template variable for clarity or add a comment explaining the trim directive, e.g.:
```
// Note: {{- .NameTitle}} uses Go template whitespace trimming to produce
// ResponseType{FieldName: ...} without a leading space.
```

### M-4: `adapter_postgres.tmpl` — `toDomain` passes `row.CreatedAt` as bare `time.Time`

**File:** `adapter_postgres.tmpl` line 166-170

```go
return domain.Reconstitute(
    domain.{{.NameID}}(row.ID.String()),
    row.Name,
    row.CreatedAt, row.UpdatedAt, deletedAt,
)
```

The sqlc `User` model has `CreatedAt time.Time` (not `pgtype.Timestamptz`). For the template's new entity, the generated sqlc model will also use `time.Time` for `created_at`/`updated_at` since the migration uses `TIMESTAMPTZ NOT NULL DEFAULT NOW()`. This is consistent with the user model. No bug.

### M-5: `domain_errors.tmpl` defines `ErrInvalidName` but entity has no method that returns it

**File:** `domain_errors.tmpl` line 11

```go
ErrInvalidName = sharederr.New(sharederr.CodeInvalidArgument, "invalid name")
```

The `domain_entity.tmpl` `ChangeName` method only returns `ErrNameRequired` (empty name check). `ErrInvalidName` is defined but never used in any generated code. This creates a dead symbol from day one.

**Recommendation:** Remove `ErrInvalidName` from the template, or add a corresponding validation path in `ChangeName` (e.g. max length check). The user module's `errors.go` does not have an equivalent dead error.

### M-6: `queries.tmpl` — missing keyset pagination index for cursor performance

**File:** `migration.tmpl` lines 10-11

The migration creates:
```sql
CREATE INDEX idx_{{.NamePlural}}_name ON {{.NamePlural}} (name) WHERE deleted_at IS NULL;
CREATE INDEX idx_{{.NamePlural}}_active ON {{.NamePlural}} (id) WHERE deleted_at IS NULL;
```

The `List` query orders by `(created_at DESC, id DESC)` and uses a cursor on `(created_at, id)`. There is no composite index on `(created_at DESC, id DESC)` that would support efficient keyset pagination. The `idx_active` index on `(id)` alone will not help for the list query.

**Recommendation:** Add the keyset index to `migration.tmpl`:
```sql
CREATE INDEX idx_{{.NamePlural}}_cursor ON {{.NamePlural}} (created_at DESC, id DESC) WHERE deleted_at IS NULL;
```

## Low Priority

### L-1: `--name` validation allows leading/trailing underscores

`_product` or `product_` pass validation (only letters and underscores, lowercase). These produce syntactically valid but semantically odd package names and identifiers like `package _product`. Go allows identifiers starting with `_` but it is unconventional.

**Recommendation:** Add a leading/trailing underscore check:
```go
if strings.HasPrefix(*name, "_") || strings.HasSuffix(*name, "_") {
    fmt.Fprintln(os.Stderr, "error: name must not start or end with underscore")
    os.Exit(1)
}
```

### L-2: No `--output-dir` / `--dry-run` flags

The generator always writes into the CWD. If run from the wrong directory (e.g. a subdirectory), all 19 files are created in incorrect relative paths. The conflict check (`os.Stat`) runs before writing but does not verify CWD is the project root.

**Recommendation:** Either check for a sentinel file (e.g. `go.mod`) before writing, or add a `--dry-run` flag that prints what would be created. A simple root check:
```go
if _, err := os.Stat("go.mod"); os.IsNotExist(err) {
    fmt.Fprintln(os.Stderr, "error: must be run from project root (go.mod not found)")
    os.Exit(1)
}
```

### L-3: Partial failure leaves orphaned files

**File:** `main.go` lines 113-133

If a template fails midway (e.g. template parse error on file 10 of 19), files 1-9 already exist on disk. The program exits with an error but does not clean up already-written files. The conflict check on the next run will fail on whichever files were created.

This is an edge case (template errors would be caught at parse time before any files are written, line 107-111), but a failed `os.MkdirAll` or `os.OpenFile` mid-run could leave partial output.

**Recommendation:** Track written files and defer a cleanup function:
```go
var written []string
defer func() {
    if r := recover(); r != nil {
        for _, f := range written {
            os.Remove(f)
        }
    }
}()
// After each successful write: written = append(written, f.out)
```

Or at minimum, document the manual cleanup step in the error message.

### L-4: `app_list.tmpl` result struct uses `Items` field, inconsistent with user module

**File:** `app_list.tmpl` line 11

```go
type List{{.NamePluralTitle}}Result struct {
    Items      []*domain.{{.NameTitle}}
    NextCursor string
    HasMore    bool
}
```

The user module's `ListUsersResult` uses `Users []*domain.User` (named after the entity). The template uses the generic `Items`. Both work, but there is now an inconsistency between the user module and all generated modules. The gRPC handler in the user module references `result.Users`, while the template handler correctly references `result.Items`.

This is not a bug — it is a conscious simplification. But if a developer looks at both modules side-by-side for pattern guidance, the difference may cause confusion.

### L-5: `domain_test.tmpl` — `Test{{.NameTitle}}_ChangeName` uses `Before` instead of `Equal`/`After`

**File:** `domain_test.tmpl` lines 43-45

```go
if entity.UpdatedAt().Before(oldUpdated) {
    t.Error("expected updatedAt to not go backwards")
}
```

`time.Now()` calls inside `ChangeName` and the test's `oldUpdated = entity.UpdatedAt()` may be identical if the clock resolution is coarse (e.g. under WSL or in CI). The test would pass even if `updatedAt` was not mutated, as `!Before` includes equal. A more reliable assertion is:

```go
if !entity.UpdatedAt().After(oldUpdated) {
    t.Error("expected updatedAt to advance after ChangeName")
}
```

Though in practice Go's `time.Now()` has nanosecond resolution on Darwin/Linux. Low-risk.

## Edge Cases Found by Scouting

1. **Snake-case plural with custom `--plural` flag**: `--name order_item --plural order_items` — `toTitle("order_items")` → `"OrderItems"`. Works correctly. `NamePlural = "order_items"`, `NamePluralTitle = "OrderItems"`. SQL table name will be `order_items` which is valid.

2. **Single-character name**: `--name x` — produces `XID`, `X` struct, `package x`. All valid Go, all valid SQL. No issue.

3. **Name collision with Go builtins**: `--name error` passes validation (all letters, lowercase) and produces `package error` which conflicts with the built-in `error` interface. Mitigation: add a blocklist check for reserved identifiers (`error`, `string`, `int`, `map`, `type`, etc.).

4. **Name collision with existing modules**: The conflict check (`os.Stat`) only checks files, not Go package names. A module named `user` that already exists would be caught by the file-exists check. A name like `grpc` or `app` would create a package in the wrong location but would not be caught — though the file path `internal/modules/grpc/` would likely not collide in practice.

5. **Template parse error exposure**: `template.ParseFS` parses all 19 templates at once. A syntax error in any template is caught before file creation begins. This is the correct order of operations and prevents partial writes for that class of error.

## Positive Observations

- Conflict-check-before-write pattern (lines 98-104) prevents accidental overwrites.
- `O_EXCL` flag on `os.OpenFile` provides a race-safe write guard.
- Embed via `//go:embed` means the binary is self-contained — no runtime template path issues.
- `toTitle` correctly handles multi-word snake_case names.
- All 5 CRUD operations are covered including soft-delete and cursor pagination.
- Generated `module.go` uses `fx.Annotate` + `fx.As` for interface binding — matches user module exactly.
- `adapter_grpc_routes.tmpl` includes `connectrpc.com/validate` interceptor from day one.
- `domain_repository.tmpl` includes `//go:generate mockgen` directive — tests will have mocks ready after `go generate`.
- Proto template includes `buf.validate` field constraints on all IDs and the name field.
- `adapter_postgres.tmpl` correctly uses `parse{{.NameTitle}}ID` (safe `uuid.Parse`) everywhere except `Create` (see H-2).
- Pagination logic (limit+1 fetch, hasMore detection, cursor encoding) is faithfully replicated.

## Unresolved Questions

1. Should `--plural` support irregular plurals that differ in structure (e.g. `--name person --plural people`)? The current `toTitle` would produce `People` which is fine, but SQL table `people` and Go type `People` would be generated. Verify this end-to-end.
2. Is there a plan to support multi-field entities via a DSL flag (e.g. `--fields "title:string,price:int"`) or is manual post-scaffold editing the intended workflow?
3. The `GoModule` is hardcoded to `github.com/gnha/gnha-services` in `main.go` (line 65). Should this be read from `go.mod` dynamically to support forks or renamed repos?
